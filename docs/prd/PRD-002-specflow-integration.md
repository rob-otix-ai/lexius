# PRD-002: Specflow Contract Compliance Integration

## Status: Draft
## Date: 2026-04-15
## Author: Robert

---

## Problem Statement

LLM-driven development drifts. Rules written in conversation context get ignored after a few iterations. Architectural boundaries (clean architecture layers, dependency direction, legislation-agnostic domain) must be enforced mechanically, not by convention.

## Solution

Integrate [specflow-cli](https://github.com/fall-development-rob/specflow-cli) as the contract compliance layer. Specflow turns architectural specifications into YAML contracts that fail the build when violated.

## Why Specflow

- **Deterministic enforcement** — contracts are rule-based, not probabilistic
- **YAML-based** — readable, version-controlled, reviewable
- **Claude Code integration** — MCP server + skills for in-editor compliance
- **CI/CD pipeline generation** — auto-generates GitHub Actions / GitLab CI from detected stack
- **Knowledge graph** — tracks compliance trends and suggests fixes over time

## Contracts Required

### Architecture Contracts

| ID | Rule | Scope |
|----|------|-------|
| ARCH-001 | Domain layer has zero infrastructure imports | `packages/core/src/domain/**` |
| ARCH-002 | Use cases depend only on domain ports, never on infrastructure | `packages/core/src/use-cases/**` |
| ARCH-003 | Infrastructure implements domain ports | `packages/core/src/infrastructure/**` |
| ARCH-004 | API layer never imports domain directly, only through use cases | `packages/api/src/**` |
| ARCH-005 | Legislation-specific logic lives only in plugin modules | `packages/core/src/legislation/**` |
| ARCH-006 | No cross-package imports except through published interfaces | `packages/*/src/**` |

### Security Contracts

| ID | Rule | Scope |
|----|------|-------|
| SEC-001 | No hardcoded secrets or credentials | `**/*.{ts,js}` |
| SEC-002 | SQL queries use parameterised statements only | `packages/db/**` |
| SEC-003 | API input validation on all endpoints | `packages/api/src/routes/**` |

### Quality Contracts

| ID | Rule | Scope |
|----|------|-------|
| QA-001 | Every use case has a corresponding test file | `packages/core/src/use-cases/**` |
| QA-002 | Seed data has validation tests | `packages/db/src/seeds/**` |
| QA-003 | No `any` type in domain layer | `packages/core/src/domain/**` |

## Integration Points

1. **`specflow init`** during project bootstrap — detects stack, generates contracts
2. **Git hooks** — `specflow enforce` on pre-commit via `.claude/settings.json` hooks
3. **CI pipeline** — `specflow enforce` as a CI step, generated by specflow's ci-builder agent
4. **Claude Code skill** — `/specflow` skill for in-editor compliance checks
5. **MCP server** — specflow's MCP tools available alongside the legal-ai MCP tools

## Workflow

```
Developer writes code
        │
        ▼
  specflow enforce  ← pre-commit hook
        │
   ┌────┴────┐
   │ PASS    │ FAIL
   │         │
   ▼         ▼
 commit    violation report
           with fix suggestions
```

## Acceptance Criteria

- [ ] `specflow init` runs cleanly against the monorepo
- [ ] All ARCH contracts pass on the initial codebase
- [ ] `specflow enforce` runs in under 5 seconds
- [ ] CI pipeline includes specflow enforcement step
- [ ] Violations block commits with actionable error messages
