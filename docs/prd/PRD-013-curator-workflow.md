# PRD-013: Curator Workflow — Live Editing for CURATED-Tier Facts

## Status: Draft
## Date: 2026-04-20
## Author: Robert

---

## Problem Statement

Lexius carries three provenance tiers on every fact: AUTHORITATIVE, CURATED, AI_GENERATED. The schema enforces `curated_by` and `reviewed_at` columns on CURATED rows (PROV-001/003), and the seed helper stamps them at build time.

What Lexius does not yet have is a way for a named domain expert to change, add, or deprecate a CURATED fact after seed. Today, "curation" means editing TypeScript seeds, re-running migrations, and rebuilding. The workflow on the schema is real; the workflow in practice is a developer workflow with a curator label on it.

This gap undermines the positioning of the product. When a reader points at "CURATED" on a Lexius answer, the honest claim should be: *a named human expert wrote or reviewed this, we can tell you who they are and when they last looked at it, and they can amend it without a code deploy.* That claim is not true today.

## Vision

A named domain expert runs `lexius-curate login`, pastes a curator-scoped API key, then opens their editor on a CURATED obligation, changes the interpretive text, saves, and sees the change reflected in every consumer (API, MCP, CLI, agent, swarm) within seconds — with a full audit trail, cross-check against verbatim law, regenerated embedding, and the option to revert any edit at any time.

Wiki-level editing (v1.2) lets the same expert do this inside Obsidian over a vault generated by `@robotixai/wikivault`, with the same guarantees.

## Users

| Persona | Need |
|---------|------|
| **Domain-expert curator** (regulatory lawyer, compliance SME) | Edit or add CURATED interpretations without opening a PR. See their edit history. Revert their own mistakes. |
| **Compliance officer consuming Lexius** | Trust that a CURATED answer reflects current expert thinking, attributed to a named individual with a recent review timestamp. |
| **Integration-maintainer (me)** | Re-run fetcher and seeds without clobbering curator work. See which curated facts went stale when verbatim law changed. |

## Scope

### In Scope (v1)

1. **One entity type only: `obligations`.** Prove the pattern end-to-end before generalising.
2. **Six capabilities:** create, edit, soft-delete (deprecate), revert, list-my-edits, view-history.
3. **Curator authentication:** paste-a-key login via `lexius-curate login --key lx_curator_...`, credentials written to `~/.config/lexius/credentials` (mode 600), profile-aware.
4. **Write guarantees on every mutation:**
   - Cross-check: numeric edits that would break `pnpm crosscheck` are rejected at edit time.
   - Re-embedding: text edits regenerate `embedding` in the same transaction.
   - Audit: one row in `curator_edits` per write, inside the same transaction.
   - Concurrency: `If-Match: <row_version>` header, 409 on mismatch.
   - Reason: every edit requires a non-empty `reason` string.
   - Tier transition: AUTHORITATIVE → immutable; AI_GENERATED → CURATED allowed; CURATED → AI_GENERATED forbidden.
   - **Anchoring:** every CURATED row carries a non-empty `derivedFrom`, and every ID in it resolves to an existing `articles` row. Curators cannot author orphan interpretations — extraction is the core, curator prose is the interpretation of it. Enforced on create and on update by `C-INT-007`.
5. **Verbatim-law drift detection:** when fetcher re-ingests an article with a changed `source_hash`, all CURATED obligations whose `derivedFrom` contains that article are flagged `needs_review = true`.
6. **Seed idempotency:** rerunning seeds over curator-edited rows is a no-op. Seed only touches rows whose `curated_by = 'seed:rob'`.
7. **Dynamic MCP tool listing:** reader-role keys see read tools only. Curator-role keys see curator tools as well. Write tools are physically absent when no curator credential is present.
8. **CLI package:** `@robotixai/lexius-curate` with `login`, `logout`, `whoami`, `obligations (list|edit|create|deprecate|history)`, `my-edits`, `revert`.

### Out of Scope (v1)

- Edit paths for penalties, deadlines, FAQ, risk-categories. Same pattern, different branch.
- Agent and MCP *write* paths. Read paths remain unchanged. Write paths land in v1.1 with two-phase propose/commit for MCP and chat-confirmation gates for the agent.
- Wikivault bidirectional editing. Designed for v1.2 once `@robotixai/wikivault` v0.1 exists.
- Browser-based curator login flow. Paste-a-key first; OAuth-style arrives with a curator admin UI later.
- Per-jurisdiction role scopes (`curator:eu`, `curator:cima`). The column is text so scopes are a zero-migration upgrade, but enforcement is out of scope for v1.
- Retro-labelling of historical `seed:rob` attributions. A dedicated CLI lives in a later branch.
- Realtime notifications (webhook, email) on curator edits.

## Product Requirements

### P0 — Must Have

1. **Migration `0005_curator_workflow.sql`** adds `api_keys.role`, `obligations.row_version`, `obligations.needs_review`, `obligations.stale_since`, `obligations.deprecated_at`, `obligations.deprecated_reason`, and the `curator_edits` table.
2. **Use cases:** `UpdateCuratedObligation`, `CreateCuratedObligation`, `DeprecateCuratedObligation`, `RevertCuratorEdit`, `ListCuratorEdits`, `MarkStaleByArticle`. All write use cases run in a single DB transaction, call the cross-check and embedding services, and write the `curator_edits` row inline.
3. **API routes** on `/api/v1/curate/*` scoped to curator role via auth middleware. Dry-run (`?dry_run=true`) supported on every write route.
4. **Shared credentials package `@robotixai/lexius-credentials`** — ~80 lines, zero deps, resolves profile → env → file in that order.
5. **CLI package `@robotixai/lexius-curate`** — full coverage of the six capabilities plus `login`/`logout`/`whoami`.
6. **Specflow contracts:** `curator_audit.yml`, `curator_auth.yml`, `curator_integrity.yml`, plus one new rule `PROV-008` on `provenance_tiers.yml` gating CURATED mutations through the use-case layer.
7. **Exit criterion test (scripted, runs against a fresh DB):** login → create obligation → edit twice → view history → revert the second edit → view history again. All six API paths exercised, all audit rows present, cross-check remains clean after every step.

### P1 — Should Have

- Fetcher emits a summary line per ingest: `N obligations flagged needs_review after <legislation> ingest`.
- CLI dry-run is the default when `$LEXIUS_CURATOR_KEY` is set but no explicit `--apply`. Reduces foot-guns.
- `/api/v1/curate/queue` returns obligations with `needs_review = true`, ordered by `stale_since`.
- CLI captures `editor_ip` and `editor_ua` automatically in audit requests.

### P2 — Nice to Have

- Key rotation: `lexius-curate refresh` exchanges a near-expiry curator key for a new one with 24h overlap.
- Structured error surface on cross-check/embedding failures (JSON error body, machine-readable codes).

## Non-Functional Requirements

- **Auditability:** every write surfaces in `curator_edits` with editor identity, action, old/new values, reason, and row-version before/after. Append-only. No delete path in application code.
- **Transactional integrity:** row update, embedding write, audit insert all live in one DB transaction. Embedding failure rolls back the row write.
- **Honesty of the label:** a CURATED row's `curated_by` must always reflect the human who most recently edited it, or `seed:rob` if never edited. A CURATED row's `reviewed_at` must always reflect the most recent edit time, or the seed time otherwise.
- **Curators cannot override mined facts.** If an `article_extracts` value is wrong, the remedy is to fix the extractor or refetch the source — not to paper over it with a curator edit. This preserves the hash-verified AUTHORITATIVE invariant and ensures every user benefits from a single fix.
- **Attacker model:** a leaked reader key cannot reach curator routes (auth middleware). A leaked curator key can read and write all obligations but cannot mutate AUTHORITATIVE data (tier gate) and cannot bypass the audit table (use-case gate enforced by contract).

## Simulation Results (from planning round)

Twenty-five edge cases modelled. Four rated as v1 blockers, ten as hazards requiring explicit handling, the remainder as design decisions or non-v1 scope.

| Blocker | Resolution in plan |
|---|---|
| Cross-check regression on numeric edits | Run cross-check synchronously inside PATCH; reject on mismatch with the mismatch list |
| Embedding staleness on text edits | Re-embed inside same transaction; fail closed |
| Lost updates under concurrency | `row_version` + `If-Match` header; 409 on mismatch |
| Split-brain row vs audit | Single DB transaction across row update + audit insert + embedding write |

| Hazard | Resolution in plan |
|---|---|
| Verbatim drift under curated interpretation | `needs_review`/`stale_since` flags, set by fetcher on `source_hash` change |
| Seed vs edit conflict on re-seed | Seed insert becomes `ON CONFLICT ... WHERE curated_by = 'seed:rob'` |
| Tier transitions | Explicit nine-cell transition matrix in use-case layer |
| Multi-curator attribution | `api_keys.owner_email` referenced by `curator_edits.editor_id` |
| Swarm finding staleness | `compliance_workspace` sessions record `source_row_versions`; stale warning on read |
| `derivedFrom` immutability on CURATED | Rejected by use-case layer (v1 is strict; relaxable later) |
| Create vs edit | POST supported from day one, not a follow-up |
| Delete semantics | Soft-delete only via `deprecated_at` + `deprecated_reason` |
| Credential handling | Credentials file mode 600, env wins for CI, CLI never logs the key |
| End-to-end honesty | Exit-criterion scripted scenario, tested in CI |
